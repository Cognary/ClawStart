# ClawStart UI 重构方案

## 1. 文档目的

这份文档用于重新定义 ClawStart 的产品形态、信息架构和前端实现边界。

目标不是“把现有页面修漂亮一点”，而是把产品彻底拆成两种完全不同的使用阶段：

1. 安装引导态
2. 安装后维护态

这两个阶段的目标、用户心智、页面内容、主操作、辅助信息密度都不一样，不能继续共用一套“什么都往页面上摆”的布局。

本方案将作为后续重构的唯一设计依据。在这个方案确认前，不继续做零散 UI 修改。

---

## 2. 当前问题总结

### 2.1 根本问题

当前页面最大的问题不是“风格不够统一”，而是产品结构错了。

现在的实现把下面这些完全不同层级的内容同时摆在一个页面上：

- 安装步骤
- 当前步骤说明
- 环境探针
- 交互式终端
- 诊断建议
- 配置编辑
- 运行中的任务
- 更新入口
- 命令日志

这会导致页面看起来像一个开发者调试面板，而不是一个“普通用户也能顺着走完”的启动器。

### 2.2 用户体验层面的具体问题

#### 首次进入时

用户第一次打开启动器，真正想知道的只有三件事：

1. 我现在在哪一步
2. 下一步该做什么
3. 这一步做完后会发生什么

但当前页面会让用户同时看到：

- 多块卡片
- 多个模块标题
- 多种按钮
- 一些当前根本不需要关心的维护能力

结果是用户不知道：

- 哪个按钮才是主按钮
- 右侧哪块才是当前应该看的区域
- 终端和日志到底是必须看的，还是高级调试用的
- 什么时候算安装完成

#### 安装完成后

安装完成后的用户诉求已经变了，不再是“跟着步骤装”，而是：

- 启动 Dashboard
- 查看服务状态
- 重跑 Doctor / Status
- 重新打开 Onboarding
- 改配置
- 看运行日志

这时继续展示“Step 1/2/3/4/5”会造成新的噪音。

也就是说：

- 安装态需要向导
- 维护态需要控制台

这两种页面应该是两套 UI，不是一套 UI 做条件渲染后硬凑出来。

### 2.3 现有实现的问题归类

#### 信息架构问题

- 页面没有清晰的一号操作区
- 页面没有明确主次内容
- 安装流程和维护工具混在同一层级
- 右侧区域承担了过多职责

#### 视觉与布局问题

- 卡片过多
- 大块空白与大块内容并存
- 同屏信息密度不稳定
- 模块之间没有形成明确阅读顺序

#### 交互问题

- 按钮多，但主按钮不够突出
- 用户很难判断当前页面的目标
- 完成一个步骤后，没有明显进入下一个阶段的“切换感”

#### 产品心智问题

- 看起来像“开发工具面板”
- 不像“启动器”
- 不像“安装向导”
- 不像“维护控制台”

## 3. 产品重构原则

### 3.1 第一原则：一屏只服务一个目标

首次安装时，一屏只服务“完成当前步骤”。

安装完成后，一屏只服务“运行和维护环境”。

### 3.2 第二原则：任何时刻只能有一个主操作

每个页面必须只有一个视觉上绝对明确的主操作按钮。

允许有次级按钮，但次级按钮只能做辅助动作，例如：

- 重新检测
- 查看日志
- 打开调试 Shell

不能和主路径竞争注意力。

### 3.3 第三原则：终端和日志是工具，不是首页主角

终端、日志、配置编辑器都应该被视为“工具层”。

它们应该在下面这些情况出现：

- 用户进入某个必须交互的步骤
- 某一步失败，需要排障
- 安装完成后用户主动切换到维护工具

它们不应该默认和引导文案并列争夺注意力。

### 3.4 第四原则：页面必须具备阶段切换感

用户应该明确感知到：

- “我现在还在安装”
- “我已经装好了”
- “我现在在维护环境”

换句话说，安装完成后，页面应该像“进入另一个产品阶段”，而不是同一堆卡片状态变了几个字。

### 3.5 第五原则：普通用户优先，开发者调试能力后置

默认优先级：

1. 让普通用户按顺序装好
2. 让装好后的用户快速启动和维护
3. 再给高级用户排障工具

不是反过来。

## 4. 目标用户与核心任务

## 4.1 用户类型

### A. 首次安装用户

特点：

- 不熟悉 OpenClaw
- 不熟悉命令行
- 只想尽快装好并能跑起来

核心任务：

- 检查环境
- 安装 CLI
- 生成配置
- 完成 Onboarding
- 通过验证
- 打开 Dashboard

### B. 已安装用户

特点：

- 已经完成首次安装
- 需要偶尔维护和排障

核心任务：

- 启动 Dashboard / Gateway
- 检查状态
- 更新 CLI
- 修改配置
- 重跑验证
- 查看日志

### C. 高级用户 / 排障用户

特点：

- 遇到 PATH、TLS、权限、端口等问题
- 需要更低层调试工具

核心任务：

- 打开调试 Shell
- 查看最近失败日志
- 使用诊断与修复动作

## 5. 新的产品结构

ClawStart 重构后应当只有两个顶层工作态。

## 5.1 顶层态 A：安装引导态

这个阶段的页面职责：

- 明确告诉用户现在在哪一步
- 给出唯一主操作
- 只在必要时展开辅助工具

这个阶段的页面本质：

- 安装向导
- 不是控制台

## 5.2 顶层态 B：维护工作台态

这个阶段的页面职责：

- 展示运行状态
- 提供常用操作
- 收纳维护工具

这个阶段的页面本质：

- 运行与维护控制台
- 不是安装向导

## 5.3 顶层态切换条件

### 进入安装引导态

满足任一条件：

- 没检测到 `openclaw`
- 配置未保存或无效
- 未完成 onboarding / 未完成 doctor / status 验证

### 进入维护工作台态

必须同时满足：

- `openclaw` 可用
- 配置有效
- workspace 已明确
- 最近一次 doctor 或 status 成功

### 回退到安装引导态

满足任一条件：

- 用户主动点击“重新进入安装引导”
- 配置失效
- 关键验证不通过

## 6. 页面总览

## 6.1 新版窗口整体骨架

窗口仍然使用固定高度，但结构重做为：

1. 左侧：阶段导航栏
2. 右侧：单一主舞台

核心区别是：

- 左侧永远是导航和状态，不是内容堆栈
- 右侧永远只有一个主任务区域，不允许多个大模块同时抢视觉中心

---

## 7. 安装引导态设计

## 7.1 页面目标

帮助用户完成首次安装，核心要求：

- 低认知负担
- 强步骤感
- 一步一步推进

## 7.2 左侧结构

安装引导态左侧固定为三段：

### A. 顶部产品信息

内容：

- 产品名 `ClawStart`
- 页面标题：`OpenClaw 安装向导`
- 一句极短说明

示例：

“按顺序完成 5 步，装好 OpenClaw 并进入 Dashboard。”

要求：

- 不要长说明
- 不要大段文案
- 不要多个提示块

### B. 中部步骤导航

只保留 5 步：

1. 检查环境
2. 安装 CLI
3. 保存配置
4. 完成 Onboarding
5. 验证并完成安装

每一步只展示：

- 序号
- 标题
- 状态

状态只允许三种：

- 未开始
- 当前步骤
- 已完成

这里不放按钮，不放详细解释，不放大段描述。

点击步骤可以查看说明，但只有“当前步骤”允许执行主操作。

### C. 底部状态摘要

只保留简短摘要：

- CLI：已安装 / 未安装
- 配置：已保存 / 未完成
- 验证：已通过 / 未通过

这部分必须压缩成极简摘要，不再出现大块卡片。

## 7.3 右侧结构

安装引导态右侧只允许出现以下三层。

### A. 顶部：当前步骤标题区

只包含：

- 当前步骤名称
- 当前步骤目的
- 完成后会发生什么

示例：

标题：
`Step 4 · 完成 Onboarding`

说明：
`这一阶段需要在应用内完成配对。完成后会进入最后验证步骤。`

要求：

- 不放别的卡片
- 不放环境摘要
- 不放配置摘要

### B. 中部：当前步骤主操作区

这是整个页面的主舞台，只能放当前步骤需要的内容。

#### Step 1 检查环境

主舞台展示：

- 是否满足环境条件
- 缺什么
- 主按钮：`重新检测`

必要时附加：

- 次按钮：`打开调试 Shell`

#### Step 2 安装 CLI

主舞台展示：

- 当前推荐安装方式
- 安装会装到哪里
- 主按钮：`开始安装`

次按钮最多一个：

- `查看安装说明`

#### Step 3 保存配置

主舞台展示：

- 表单化配置，而不是原始 JSON 大编辑器

必须提供的字段：

- workspace 路径
- gateway host
- gateway port

高级配置处理方式：

- 默认折叠在“高级设置”
- 不在第一屏展开

主按钮：

- `保存并继续`

次按钮：

- `恢复默认模板`

#### Step 4 完成 Onboarding

主舞台展示：

- 明确说明为什么要配对
- 一个主按钮：`开始 Onboarding`
- 启动后，把内嵌终端替换进主舞台

这里的关键要求：

- 终端只在这个步骤成为主角
- 终端出现前不提前占版面

#### Step 5 验证并完成安装

主舞台展示：

- `Doctor / Status` 验证结果
- 如果通过，显示大号完成状态
- 主按钮：`完成并进入工作台`

如果失败：

- 显示最重要的一条问题
- 给出一个主修复动作
- 允许展开“查看更多诊断”

### C. 底部：辅助区

安装引导态的辅助区必须是折叠式或切换式，不允许默认长列表。

辅助区仅允许包含：

- 最近输出
- 问题排查
- 运行任务

默认只展开一个：

- 当前步骤最相关的那个

例如：

- 安装阶段默认显示“最近输出”
- 验证阶段默认显示“问题排查”

## 8. 维护工作台态设计

## 8.1 页面目标

帮助用户在安装完成后进行日常启动、检查和维护。

这个页面应该像“控制台”，不是“向导”。

## 8.2 左侧结构

维护态左侧不再展示安装步骤列表作为主内容。

左侧改为四段：

### A. 顶部标题

- 页面标题：`OpenClaw 工作台`
- 副标题：`已完成安装，可直接启动和维护`

### B. 服务状态

展示三个核心状态：

- Dashboard：运行中 / 未启动
- Gateway：运行中 / 未启动
- CLI：版本号

### C. 快捷入口

只保留高频操作：

- 打开 Dashboard
- 启动 Gateway
- 运行 Doctor
- 查看 Status

### D. 入口切换

可切回：

- `重新进入安装引导`

用于配置失效后重新跟步骤处理。

## 8.3 右侧结构

维护态右侧必须是“单主舞台 + 工具切换”。

### A. 顶部总览

总览只展示：

- 当前环境可用性
- 当前 workspace
- 当前 dashboard 地址
- 最近一次验证状态

### B. 主操作条

主操作条只保留 1 个主按钮和 2 到 3 个次按钮。

推荐：

- 主按钮：`打开 Dashboard`
- 次按钮：`启动 Gateway`
- 次按钮：`运行 Doctor`
- 次按钮：`更新中心`

### C. 维护面板

采用 tab 切换，只保留四个面板：

1. 诊断
2. 终端
3. 配置
4. 日志

任何时刻只显示一个。

#### 面板 1：诊断

显示：

- 当前最重要的问题
- 修复建议
- 一键修复动作

不显示：

- 大量历史日志
- 无关摘要卡片

#### 面板 2：终端

显示：

- 应用内 Onboarding 入口
- 调试 Shell 入口
- 当前终端会话

#### 面板 3：配置

默认展示表单视图。

需要一个“查看原始 JSON5”切换，但默认必须是表单，不是文本编辑器。

配置面板分层：

- 基础配置
- 网络配置
- 高级配置

#### 面板 4：日志

显示：

- 最近命令输出
- 可切换任务日志
- 可复制错误信息

不显示：

- 所有历史日志堆在首页

## 9. 关键交互规则

## 9.1 主按钮规则

每个页面必须有且仅有一个主按钮。

### 安装引导态

主按钮始终是：

- 当前步骤的推进动作

例如：

- 开始安装
- 保存并继续
- 开始 Onboarding
- 运行验证
- 完成并进入工作台

### 维护工作台态

主按钮默认为：

- 打开 Dashboard

如果 Dashboard 未就绪，则主按钮变为：

- 启动 Dashboard

## 9.2 次按钮规则

次按钮必须满足：

- 数量少
- 服务于主任务
- 不与主按钮竞争视觉焦点

## 9.3 反馈规则

每个动作都必须有完整反馈周期：

1. 开始中
2. 运行中
3. 成功
4. 失败
5. 下一步是什么

例如安装成功后，不只显示：

`安装成功`

而应该显示：

`OpenClaw CLI 已安装。下一步：保存基础配置。`

## 9.4 错误展示规则

错误不能直接把原始 stderr 整屏扔给用户。

错误展示分三层：

### 第一层：一句人话总结

例如：

- 找不到 `openclaw` 命令
- 配置文件解析失败
- 端口已被占用

### 第二层：推荐动作

例如：

- 一键修复 PATH
- 恢复默认配置
- 释放端口并重试

### 第三层：技术细节

例如：

- 原始日志
- 错误片段
- 打开调试 Shell

## 10. 文案策略

## 10.1 安装态文案风格

要求：

- 短句
- 指令明确
- 只说当前步骤需要知道的事

避免：

- 大段说明
- 解释过度
- 同时提多个工具

## 10.2 维护态文案风格

要求：

- 状态导向
- 操作导向
- 低噪音

避免：

- 继续使用“Step 1/2/3/4/5”作为视觉主角

---

## 11. 视觉层级要求

## 11.1 安装态视觉目标

安装态应该让用户感觉是：

- 一个安装向导
- 一个流程化工具

不能像：

- IDE 面板
- 调试器
- DevOps 控制台

### 安装态视觉特征

- 左侧步骤列表清晰但简洁
- 右侧主卡片大而明确
- 终端只在 Step 4 才占大面积
- 日志默认收起来
- 高级能力后置

## 11.2 维护态视觉目标

维护态应该让用户感觉是：

- 一个日常控制台
- 可以快速启动和检查环境

### 维护态视觉特征

- 总览简洁
- 主操作明显
- 工具区使用 tab
- 页面高度稳定

## 13. 状态机设计

UI 不能再靠零散布尔值拼接，需要明确状态机。

## 13.1 安装阶段状态机

建议状态：

- `checking_environment`
- `ready_to_install`
- `installing_cli`
- `ready_to_configure`
- `saving_config`
- `ready_for_onboarding`
- `running_onboarding`
- `ready_to_verify`
- `verifying`
- `setup_complete`
- `setup_blocked`

## 13.2 维护阶段状态机

建议状态：

- `idle`
- `dashboard_running`
- `gateway_running`
- `doctor_running`
- `config_dirty`
- `repair_in_progress`
- `update_available`

## 13.3 UI 切换规则

页面显示应优先依据状态机，而不是多个散乱条件判断。

例如：

- `running_onboarding` 时，右侧主舞台必须是终端
- `ready_to_verify` 时，右侧主舞台必须是验证面板
- `setup_complete` 后默认切到维护态

## 16. 验收标准

## 16.1 安装态验收标准

一个完全不懂 OpenClaw 的用户打开后，必须能在 10 秒内回答：

1. 我现在在哪一步
2. 下一步点哪里
3. 完成这一步后会进入什么阶段

页面必须满足：

- 右侧只存在一个明确主操作区
- 非当前步骤工具不抢主视觉
- 终端和日志不会默认铺满页面

## 16.2 维护态验收标准

一个已经安装过的用户打开后，必须能在 5 秒内回答：

1. Dashboard 是否能打开
2. 服务是不是正在运行
3. 配置去哪改
4. 日志去哪看

页面必须满足：

- 不再以“安装步骤”为核心
- 不再上下堆叠多个大型工具块
- 维护工具必须可切换，不同时铺开

## 16.3 技术验收标准

- `App.tsx` 不再承担大部分 JSX
- 安装态和维护态各自拥有独立 shell
- 不再依赖大量散乱布尔值控制 UI

## 19. 我对这次重构的明确判断

当前问题不是“样式没调好”，而是产品被做成了一个混合面板。

正确方向不是继续局部修补，而是：

- 把安装引导态单独做成真正的向导
- 把安装后维护态单独做成真正的工作台
- 用状态机驱动，而不是让所有模块同时在页面上争位置

如果按这份方案做，ClawStart 才会开始像一个“普通人能用的 OpenClaw 启动器”，而不是一个“把内部功能都暴露出来的开发者面板”。
